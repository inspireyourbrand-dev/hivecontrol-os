# Fork Strategy & Upstream Sync

## Overview

HiveClaw maintains a **soft fork** of OpenClaw, allowing us to:
- Stay synchronized with upstream improvements
- Add HivePowered-specific features and customizations
- Gracefully merge updates without losing custom code

This document outlines our approach to maintaining the fork and syncing with upstream.

---

## Directory Structure

```
HiveControl OS/
├── upstream/              # OpenClaw (soft fork source)
│   └── .git/             # Full git history tracking
├── gateway/              # HiveClaw customizations & integration layer
├── screens/              # Custom HiveControl UI screens
├── agents/               # Agent workspace (runtime, not versioned)
├── scripts/              # Maintenance & deployment automation
├── docs/                 # This documentation
├── branding/             # HivePowered theme & styling
└── hiveclaw.config.json  # Configuration (generated by install.sh)
```

### What Comes from OpenClaw

The `upstream/` directory contains the complete OpenClaw gateway implementation:
- WebSocket server
- Core routing logic
- Agent communication protocol
- Build infrastructure (package.json, node_modules)

We **do not modify** OpenClaw's core files directly. Instead, we:
1. Use git hooks and overlays for customization
2. Extend via configuration (hiveclaw.config.json)
3. Serve HiveControl assets separately

### What We Customize

Files **owned by HiveClaw** and not synced with upstream:

```
HiveControl OS/
├── gateway/serve/        # Served assets (copied from HiveControl + upstream build)
├── screens/              # 8 custom UI screens
├── index.html            # Custom entry point
├── ws-client.js          # Custom WebSocket client
├── branding/             # CSS themes, logos, fonts
├── scripts/              # Automation scripts
├── docs/                 # Documentation
└── agents/               # Runtime agent workspace
```

## Sync Cadence

### Regular Syncs (Weekly)

Run the automated sync script to pull upstream improvements:

```bash
scripts/sync-upstream.sh merge origin main
```

**Frequency**: Every Monday morning (recommended)

**Behavior**:
- Fetches latest from `openclaw/openclaw` main branch
- Merges non-breaking changes automatically
- Reports what's changed
- Runs build verification
- Stops on conflicts (requires manual resolution)

### Breaking Changes (Quarterly Review)

Every 3 months, review upstream for major API changes:

1. Check OpenClaw changelog
2. Assess impact on HiveControl integration layer
3. Update documentation if needed
4. Plan migration if required

### Force Sync (Emergency)

If upstream has critical fixes needed immediately:

```bash
cd upstream
git fetch origin main
git rebase origin/main
npm run build
cd ..
```

Then verify HiveControl still works.

## Handling Merge Conflicts

### Common Conflicts

**Scenario**: Upstream modified a file HiveControl also customized

**Resolution Strategy**:

1. **When OpenClaw modifies package.json, scripts, or build config**
   - Accept OpenClaw's changes (use `--theirs`)
   - Re-apply HiveControl customizations afterward
   - Update hiveclaw.config.json if needed

2. **When OpenClaw modifies gateway core files**
   - Understand the change fully before merging
   - Test thoroughly in HiveControl
   - Document any compatibility issues

3. **When HiveControl modifies OpenClaw files**
   - This should be rare (indicates missing abstraction)
   - Create an issue to extract customization into config
   - Plan to remove direct modification on next major version

### Conflict Resolution Process

```bash
# During conflict (sync script will pause here)
cd upstream

# View conflicts
git status
git diff

# Resolve using your editor
# Then stage and complete merge
git add <resolved-files>
git commit -m "Merge upstream: [description of what you resolved]"

# Return to root and verify
cd ..
scripts/build-hivecontrol.sh
```

## What to Keep Upstream vs. Downstream

### Keep Upstream (Don't Fork)

- Core gateway (WebSocket server)
- Agent protocol implementation
- Basic routing logic
- Build system (if unmodified)
- Dependencies (package.json versions)
- Core tests

**Rationale**: These benefit from upstream improvements and we have no HivePowered-specific needs.

### Keep Downstream (Fork)

- UI screens (all 8 custom screens)
- Branding & theme (CSS, logos, fonts)
- Custom WebSocket client extensions
- HiveControl-specific configuration
- Documentation
- Deployment scripts

**Rationale**: These are HivePowered-specific and would pollute upstream.

### Hybrid (Minimal Fork, Maximum Config)

- Agent workspace structure
- Feature flags
- Logging & telemetry setup
- Server configuration

**Approach**: Define in `hiveclaw.config.json` to avoid modifying upstream.

## Integration Points

### How HiveControl Customizations are Loaded

1. **On Startup** (install.sh):
   ```
   OpenClaw installed → HiveControl assets copied → Config created
   ```

2. **At Runtime**:
   ```
   OpenClaw starts → Serves from gateway/serve/ → HiveControl UI loaded
   ```

3. **WebSocket Protocol**:
   ```
   OpenClaw WS server ← HiveControl ws-client.js (custom)
   ```

### Configuration-Driven Customization

`hiveclaw.config.json` controls:

```json
{
  "gateway": {
    "serve": "./gateway/serve",      // Custom asset directory
    "port": 3000,
    "wsPort": 3001
  },
  "features": {
    "enableTypeScript": true,
    "enableESLint": false
  },
  "sync": {
    "mergeStrategy": "merge",
    "autoVerify": true
  }
}
```

This keeps OpenClaw's code unchanged while allowing HiveControl customization.

## Emergency Procedures

### If Upstream is Broken

1. Pause auto-syncs
2. Investigate upstream status
3. Consider reverting to known-good commit:
   ```bash
   cd upstream
   git reset --hard <known-good-commit>
   npm install && npm run build
   ```
4. Wait for upstream fix
5. Resume syncs

### If HiveControl Breaks After Sync

1. Identify which upstream change caused it
2. Create a local patch in `gateway/` without modifying `upstream/`
3. Document the incompatibility
4. File issue with OpenClaw or plan workaround
5. Continue using patched version until upstream is fixed

### If We Need Backward Compatibility

Example: OpenClaw changed WebSocket protocol but HiveControl agents depend on old format.

**Solution**:
1. Implement compatibility layer in `ws-client.js`
2. Support both old and new protocols
3. Gradually migrate agents
4. Remove compatibility shim in next major version

## Documentation

When syncing, update these files if behavior changes:

- `ARCHITECTURE.md` - If data flow changes
- `CONTRIBUTING.md` - If development setup changes
- `.sync-logs/` - Automatic logs of every sync

Before major upstream merges, document:
- What changed upstream
- How it affects HiveControl
- Any manual steps required post-sync

## Version Tracking

HiveClaw version is independent of OpenClaw:

- HiveClaw: `0.1.0` (defined in hiveclaw.config.json)
- OpenClaw upstream: Whatever commit we're synced to

Track OpenClaw commit in git:
```bash
cd upstream && git rev-parse HEAD > ../.openclaw-ref
```

This file (`.openclaw-ref`) documents which OpenClaw version we're running.

## Summary

| Aspect | Strategy |
|--------|----------|
| **Sync Frequency** | Weekly (Monday mornings) |
| **Merge Strategy** | Merge (preserves history) |
| **Conflict Handling** | Manual (safe, no force) |

| **Customization** | Via config, not code modification |
| **Documentation** | Updated per sync if needed |

For questions or to propose changes to this strategy, open an issue or contact the maintainers.
